在学习了面向对象和集合框架之后，你写的程序依然是“单线程”的。这意味着，你的代码就像是一条单行道，所有的任务只能排着队一件一件去执行。

如果我们要开发的是一个高并发的后端系统（比如秒杀商城），单线程绝对会被瞬间击垮。这时候，我们就需要引入 Java 后端最迷人也是最危险的领域——**多线程与并发编程**。

## 1. 并发带来的“灾难”：超卖现象

假设我们有一个商品库存 `stock = 1`。
当有两个用户（线程A 和 线程B）同时点击购买时，会发生什么？

1. 线程A 检查库存：`stock > 0`，条件成立。
2. **就在这微秒之间**，CPU 切换到了线程B。线程B 检查库存：此时由于线程A 还没来得及扣减，`stock` 依然大于 0，条件成立。
3. 线程A 执行扣减：`stock = 0`。
4. 线程B 也执行扣减：`stock = -1`。

这就是经典的**并发安全问题**（超卖）。如果这是在卖茅台，公司可能要赔得底裤都不剩。

## 2. 传统解药：`synchronized` 同步锁

为了解决这个问题，Java 给出了最简单粗暴的解药：锁（Lock）。

```java
public synchronized void buy() {
    if (stock > 0) {
        stock = stock - 1;
    }
}
```

加了 `synchronized` 关键字后，这个方法就像是一个只能进一个人的**公共厕所**。
线程A 进去后直接把门反锁，线程B 只能在外面干等着，直到 线程A 执行完出来把锁解开。这就完美地解决了数据错乱的问题。

## 3. 性能杀手与高阶玩法（JUC）

但是，`synchronized` 太重了。如果一万人同时抢购，9999 人都在排队等待锁释放，系统的吞吐量会低得令人发指。

于是，大神 Doug Lea 为 Java 编写了一个封神的包：`java.util.concurrent`（简称 **JUC**）。
JUC 提供了一系列核武器级别的并发工具：

*   **CAS 与 Atomic 类**：我不加锁，我只在修改的那一刻比较一下旧值对不对（乐观锁思想），极其轻量。
*   **ConcurrentHashMap**：当你多线程操作 Map 时，绝不能用普通的 `HashMap`（死循环警告）。`ConcurrentHashMap` 采用分段锁/细粒度锁，既保证安全又保证极高的并发读写。
*   **线程池（ThreadPoolExecutor）**：创建线程是非常昂贵的操作系统级操作。线程池就像是一个工厂，预先招好一批工人（线程），任务来了直接分配给空闲的工人，干完活不辞退，留着接下一个任务。这极大地降低了系统开销。

## 结语

多线程是一把锋利的双刃剑。

它能充分榨干多核 CPU 的性能，支撑起成千上万 QPS 的高并发架构；但同时，死锁、可见性、指令重排等底层幽灵，也会让你的 Bug 难以复现、防不胜防。

作为全栈开发，我们并不一定要手写多么复杂的并发控制，但在面对分布式锁（如 Redis 实现）和高并发秒杀系统时，深刻理解单机多线程的底层原理，是你拥有“架构师思维”的必经之路。
