在写后端业务逻辑时，我们每天都在干什么？无非就是把数据库里查出来的数据，**存起来、排个序、过滤一下、再发给前端**。

如果把数据比作物品，那么**集合框架（Collections Framework）**就是 Java 为我们提供的各种“收纳盒”。用错了收纳盒，不仅找东西慢（性能差），还容易装错（引发 Bug）。

今天我们就来盘点 Java 集合框架中的“御三家”：`List`、`Set` 和 `Map`。

## 1. List：排队买票的队伍
**特点：有序、可重复。**

`List` 就像是排队买票的队伍，每个人都有一个编号（索引从 0 开始），你可以允许队伍里有长得一模一样的双胞胎（可重复）。

我们最常用的有两个实现类：
*   **`ArrayList`**：底层是**数组**。优点是“随机查找快”（通过下标瞬间找到目标），缺点是“中间插入/删除慢”（因为后面的元素全都要跟着挪位置）。
*   **`LinkedList`**：底层是**双向链表**。优点是“增删快”（只需要改一下前后指针），缺点是“查找慢”（必须从头顺藤摸瓜找过去）。

**实战建议**：99% 的场景下，直接无脑用 `ArrayList` 就对了，因为业务中通常是查询多、中间插入少。

## 2. Set：不接受重复的傲娇俱乐部
**特点：无序（部分实现有序）、不可重复。**

`Set` 的核心使命就是**去重**。当你要统计一篇英文文章里出现过多少个不同的单词时，`Set` 就是最好的武器。

*   **`HashSet`**：最常用，速度极快，但不保证存放顺序。它的底层其实是一个阉割版的 `HashMap`（只用它的 Key）。
*   **`TreeSet`**：会对存入的元素自动进行排序，底层是红黑树。

## 3. Map：查字典的神器
**特点：键值对（Key-Value）映射，Key 不可重复。**

`Map` 其实并没有继承 `Collection` 接口，它是自成一派的。它就像是一本字典，你通过“偏旁部首（Key）”就能瞬间查到“对应的字（Value）”。

在 Java 面试中，**`HashMap` 的底层原理几乎是必考题**，没有之一。

### HashMap 为什么那么快？
1.  当你存入一个键值对（比如 `put("张三", 100)`）时，HashMap 会先对 Key ("张三") 进行 Hash 计算，算出一个固定的数组下标。
2.  然后直接把 Value 放进那个数组格子里。这就是为什么它的查找时间复杂度能逼近神乎其技的 `O(1)`。
3.  如果两个不同的 Key 算出了相同的下标（也就是所谓的**哈希冲突**），HashMap 会在这个格子里拉出一条“链表”。在 JDK 8 之后，如果链表太长（大于 8），还会自动转成“红黑树”来保证查询速度。

## 结语

集合框架是 Java 基础中最实用、也最能体现底层数据结构之美的地方。

在使用这些集合时，永远记住**泛型（Generics）**的重要性。写下 `List<String> list = new ArrayList<>();`，让编译器帮你检查类型错误，这是静态语言相比于动态语言最大的幸福感来源之一。
